Product Requirements Document (PRD): YSC St. Dominic Catholic ChurchProject Name: YSC St. Dominic PlatformOrganization: Youth Serving Christ (YSC), St. Dominic Catholic Church, St. Theresa Kalimoni ParishLead Developer: Wilfred KimuraTech Stack: React, Node.js (Express), PostgreSQL, Clerk, Cloudinary, Resend1. Executive SummaryThe YSC St. Dominic platform is a centralized web ecosystem designed to foster community engagement, streamline information sharing, and archive the spiritual journey of the Youth Serving Christ group. It provides a public hub for members and a robust administrative suite for leadership to manage content and track community growth.2. User Roles & PermissionsRolePermissionsGuestView public home page, blog summaries, and upcoming events.Registered UserFull access to blog content, gallery, profile management, and community engagement features (comments/prayer wall).AdminFull system access, content moderation, user management, and analytics dashboard.3. Functional Requirements3.1. Authentication & Profile ManagementClerk Integration: Seamless registration and login.Identity Management: Automatic synchronization of Clerk IDs with the local PostgreSQL users table.Member Profiles: Editable fields for bio, full name, and profile imagery.Security: Support for soft deletes on user accounts to maintain database integrity.3.2. Content & Event ManagementRich Text Blog: Implementation of TipTap for JSON-based content creation, allowing for high-fidelity rendering of reflections and testimonies.Event System: Management of upcoming and past events with a primary focus on automated member outreach.Analytics: Tracking of unique views per blog post via a dedicated post_views table (tracking ID/IP for deduplication).3.3. Media ManagementOptimized Gallery: Cloudinary-hosted image management using f_auto,q_auto tags for automated bandwidth optimization and format delivery.Categorization: Logical grouping of media into Albums with support for batch uploads and mass metadata updates.3.4. Community EngagementReal-time Notifications: Immediate email dispatch via Resend upon the creation of new events.Interaction: Discussion modules for blog posts and a digital "Prayer Wall" for member requests.Social Integration: OpenGraph metadata implementation for rich link previews on WhatsApp and social media.3.5. Administrative SuiteContent Control: CRUD operations for all site modules (Blog, Events, Leaders, Gallery).Trash Management: A "Soft Delete" recovery system allowing for the restoration of accidentally removed data.Data Insights: High-level counters for total membership, aggregate views, and event engagement.4. Technical Architecture4.1. Database Schema (PostgreSQL)Core Tables: users, blog_posts, events, albums, gallery_images, comments, prayer_requests.Analytics Table: post_views (Relational link to blog_posts).Soft Delete Logic: Every table includes a deleted_at timestamp.4.2. File & Email ServicesStorage: Cloudinary (Primary storage for all binary assets).Email: Resend (API-driven transactional emails).Transformation: Dynamic URL manipulation for on-the-fly image resizing and optimization.5. Non-Functional RequirementsPerformance: Implementation of lazy loading and skeleton screens to mitigate "cold start" delays on free-tier hosting (Render).SEO: Dynamic meta-tag injection for all blog and event slugs.Scalability: Email logic supports batching/chunking to handle community growth beyond Resend's single-call limits.Compliance: Adherence to data privacy principles by limiting local PII storage and utilizing Clerk for credential security.
